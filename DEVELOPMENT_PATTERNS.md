# DEVELOPMENT_PATTERNS.md - 성공/실패 개발 패턴 가이드

## 🎯 성공적인 개발 패턴

### 1. 아이템 시스템 개발 성공 사례 (2025-01-29)

#### ✅ 성공한 접근 방식
- **체계적 디버깅**: 각 메서드마다 단계별 로그 추가하여 정확한 실패 지점 파악
- **기존 작동 코드 분석**: 직업 스킬의 `inventory.HaveItem("$item_trophy_eikthyr")` 방식을 참고
- **Valheim 내부 명명 규칙 준수**: `"Wood"` → `"$item_wood"` 변경으로 시스템 호환성 확보
- **통합된 아이템 검증**: `dropPrefab.name`과 `shared.name` 모두 체크하는 방식

#### 📊 성공 요인 분석
1. **기존 시스템 참조**: 작동하는 직업 스킬 코드를 분석하여 패턴 발견
2. **명명 규칙 통일**: Valheim 내부 `$item_` 형식 사용으로 일관성 확보
3. **포괄적 로깅**: 문제 지점을 정확히 식별할 수 있는 디버깅 시스템
4. **단계적 수정**: 한 번에 모든 것을 바꾸지 않고 핵심 문제부터 해결

### 2. UI 통합 성공 사례

#### ✅ 성공한 접근 방식
- **기존 메서드 활용**: `SkillTreeManager.CanLearnProductionSkill` 사용
- **공개 메서드 생성**: `TryUnlockProductionSkill`을 public으로 변경
- **일관된 검증 시스템**: UI와 백엔드가 같은 검증 로직 사용

## ❌ 실패한 개발 패턴

### 1. 아이템 시스템 개발 실패 사례 (2025-01-29)

#### ❌ 실패한 접근 방식
- **잘못된 메서드 호출**: 존재하지 않는 `SkillTree.ItemManager.CanUnlockSkill` 호출
- **표준 API 가정**: `inventory.CountItems("Wood")`가 스택을 자동 집계할 것이라는 가정
- **단일 명명 규칙**: Valheim 표준이 아닌 `"Wood"` 형식 사용
- **분리된 검증 시스템**: UI와 백엔드가 다른 검증 로직 사용

#### 📊 실패 요인 분석
1. **API 가정**: Valheim API 동작 방식을 확인하지 않고 가정
2. **명명 규칙 불일치**: 게임 내부 표준과 다른 명명 규칙 사용
3. **불충분한 로깅**: 문제 지점을 정확히 파악할 수 없는 로깅 시스템
4. **시스템 분리**: UI와 백엔드 로직 불일치로 인한 혼란

### 2. 빌드 오류 패턴

#### ❌ 자주 발생하는 실패
- **메서드 가시성**: private 메서드를 public에서 호출 시도
- **반환 타입 불일치**: void 반환값을 bool로 처리 시도
- **네임스페이스 충돌**: 클래스명 중복으로 인한 모호성

3. **일관된 명명**: 프로젝트 전체에서 일관된 명명 규칙 사용
4. **메서드 분리**: 복잡한 로직은 작은 메서드로 분할
